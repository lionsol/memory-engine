# Console Annotation / Reports Handoff Smoke Test

## Goal

This smoke test verifies the read-only GUI handoff between `/reports` and `/annotations` for archived raw-log rescue review work.

The expected path is:

1. Inspect reports from `/reports`.
2. Open candidate or review queue JSONL in `/annotations`.
3. Optionally resume with the latest labels JSONL.
4. Export labels and browser-local QC JSON from `/annotations`.
5. Inspect the generated QC and alignment reports from `/reports`.

## Safety Boundary

This workflow is read-only except for browser-local file downloads initiated by the user.

It must not:

- upload labels to the server
- write DB
- mutate memory files
- apply lifecycle actions
- unarchive records
- update categories
- delete records
- quarantine records
- reinforce memories
- call an LLM
- add a new server mutation route

All server reads must go through the existing allowlisted reports API:

```text
/api/reports/file?name=<report>
```

## Required Report Families

The smoke path should use whichever files exist locally under `reports/`, but the GUI should recognize these families:

- `annotation_candidates` JSONL
- `annotation_labels` JSONL
- `annotation_local_qc_report` JSON
- `archived_raw_log_rescue_combined_report` JSON
- `archived_raw_log_rescue_review_queue` JSONL
- `archived_raw_log_rescue_review_queue_label_report` JSON
- `auto_recall_turn_gold_set_replay` JSON

For preview-capable rescue families, latest/default selection must prefer structured files over Markdown companions:

- combined rescue report: `.json`
- manual review queue: `.jsonl`
- review queue label report: `.json`

## `/reports` Smoke Steps

1. Open `/reports`.
2. Confirm latest cards render and are clickable.
3. Click the latest archived raw-log rescue combined report card.
4. Confirm `Archived Raw-log Rescue Combined Preview` renders structured scoring metrics.
5. Click the latest archived raw-log rescue review queue card.
6. Confirm `Review Queue Preview` renders queue counts, priority range, reason distribution, bucket distribution, prediction distribution, manual flags, and risk signals.
7. Confirm the report detail for a queue JSONL shows `Open in Annotations`.
8. If an `annotation_labels` JSONL exists, confirm the same detail also shows `Open with Latest Labels`.
9. Click a local QC report if present and confirm `Annotation QC Preview` renders coverage and distribution summaries.
10. Click a review queue label report if present and confirm `Review Queue Label Preview` renders aligned labels, blockers, unlabeled samples, and distributions.

## `/annotations` Smoke Steps

1. Open `/annotations` directly.
2. Load a whitelisted candidate or queue JSONL from `Available Candidate Reports`.
3. Confirm the page shows samples and progress counts.
4. Confirm `Current Deep Link` appears after server-loaded candidate/queue JSONL.
5. Click `Copy Link` and confirm the page reports success, or a manual-copy fallback appears.
6. Load a whitelisted labels JSONL from `Available Label Reports`.
7. Confirm label import is identity-aligned and reports skipped counts for invalid, not-in-candidates, identity mismatch, and empty labels.
8. Confirm `Current Deep Link` updates to include the labels report.
9. Export labels JSONL.
10. Export browser-local QC JSON.

## Deep Link Smoke Steps

Use the URLs generated by `/reports` or `/annotations`:

```text
/annotations?candidate=<candidate-or-queue.jsonl>
/annotations?candidate=<candidate-or-queue.jsonl>&labels=<labels.jsonl>
/annotations?candidate_report=<candidate-or-queue.jsonl>&label_report=<labels.jsonl>
```

Expected behavior:

- candidate or queue JSONL loads first
- labels load only after candidate load succeeds
- candidate load failure stops labels import
- non-JSONL candidate reports are rejected
- non-JSONL label reports are rejected
- local browser files do not generate server deep links

## Non-Goals

This smoke test does not approve automated cleanup. In particular:

- QC reports do not authorize deletion.
- Label reports do not authorize unarchive.
- `preferred_action` is advisory only.
- `auto_recall_eligible` labels do not directly change runtime recall gates.
- Manual review remains required before any lifecycle action.
